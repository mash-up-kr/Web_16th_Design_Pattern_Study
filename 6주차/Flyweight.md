# 플라이웨이트 패턴

- 플라이웨이트 패턴은 비슷한 객체를 대량으로 만들어야 할 때 메모리를 절약할 수 있게 해주는 유용한 패턴입니다.

## 예시

- 책을 추가할 수 있는 앱이 있다고 가정해보겠습니다.
  - 모든 책은 title, author, isbn 속성을 가지고 있습니다.
  - 하지만 도서관에서는 보통 동일한 책을 여러권 가지고 있습니다.
- 이때 동일한 책에 대해 새로운 클래스를 매번 생성하는 방식 보다는, 하나의 책을 의미하는 `Book` 클래스의 인스턴스를 여러 개 만들려고 합니다.
  ```ts
  class Book {
    constructor(title, author, isbn) {
      this.title = title
      this.author = author
      this.isbn = isbn
    }
  }
  ```
  - 아래 코드에서는 기 등록된 ISBN 이라면 새로운 Book 인스턴스를 만들 필요가 없는 구조이기 때문에, 먼저 해당 책이 이미 존재하는지를 체크합니다.
    > *ISBN (International Standard Book Number)**은 전 세계에서 출판되는 도서에 부여되는 국제 표준 도서 번호입니다.
    > 간단히 말하면, 도서의 "주민등록번호" 같은 역할을 하는 고유 식별 번호라고 생각하시면 됩니다.
  - 만약 등록되지 않은 ISBN 이라면 새로운 책 인스턴스를 만들고 이를 리스트에 등록해줍니다.
  ```ts
  const createBook = (title, author, isbn) => {
    const existingBook = books.has(isbn) // 여기에서 체크

    // 존재하는 경우
    if (existingBook) {
      return books.get(isbn)
    }

    // 없는 경우에는 새로 만들어주기
    const book = new Book(title, author, isbn)
    books.set(isbn, book)

    return book
  }
  ```
  - 하지만 앞서 언급한대로 도서관은 동일한 책을 여러 권을 보유할 수 있는 특징이 존재합니다.
  - 이를 위해 같은 책을 여러 권 추가할 수 있는 addBook 함수를 만들어보겠습니다.
    - 해당 함수에서는 책 인스턴스를 새로 만들거나 이미 존재하는 경우를 고려하여 이전에 구현한 createBook 함수를 호출해야 합니다.
  - 사본의 갯수를 추적하기 위해 bookList 배열을 만들어 도서관의 모든 책을 포함시키겠습니다.
    ```ts
    const bookList = []

    const addBook = (title, author, isbn, availability, sales) => {
      const book = {
        ...createBook(title, author, isbn),
        sales,
        availability,
        isbn,
      }

      bookList.push(book)
      return book
    }

    // 사용 예시
    addBook('Harry Potter', 'JK Rowling', 'AB123', false, 100)
    addBook('Harry Potter', 'JK Rowling', 'AB123', true, 50)
    addBook('To Kill a Mockingbird', 'Harper Lee', 'CD345', true, 10)
    addBook('To Kill a Mockingbird', 'Harper Lee', 'CD345', false, 20)
    addBook('The Great Gatsby', 'F. Scott Fitzgerald', 'EF567', false, 20)
    ```
    - 이제 사본을 추가할 때 마다 Book 인스턴스를 새로 만드는 대신 이미 존재하는 책일 경우 해당 인스턴스를 재사용하도록 구현하였습니다.
    - 사용 예시를 확인해보면 5개의 책을 추가했지만 Book의 인스턴스는 3개만 생성되었습니다.

## 정리

- 플라이웨이트 패턴을 사용하면 대량의 객체를 만들어 낼 때 메모리를 많이 사용하는 문제를 해결할 수 있습니다.
- 메모리 사용량을 최소화할 수도 있습니다.
- 하지만 자바스크립트에서 프로토타입 상속을 통해서도 유사한 효과를 낼 수 있다보니 해당 패턴은 자바스크립트 생태계에서 크게 중요하지 않게된 부분도 있습니다.
> 사실 여기까지만 보면 무슨 내용인지 잘 모르겠네요

## 책에서 가져오는 내용

### 플라이웨이트 패턴이란?

- 플라이웨이트 패턴은 반복되고 느리고 비효율적으로 데이터를 공유하는 코드를 최적화하는 구조적 해결 방법입니다.
- 연관된 객체끼리 데이터를 공유하면서 애플리케이션의 메모리를 최소화하는 목적을 가지고 있습니다.
- 플라이웨이트 패턴의 데이터 공유 방식은 여러 비슷한 객체나 데이터 구조에서 공통으로 사용되는 부분만을 하나의 외부 객체로 내보내는 것으로 이루어집니다.
- 각 객체에서 데이터를 저장하는 방식보다는 하나의 의존 외부 데이터에 모아서 저장할 수 있습니다.
> 책 내용 그대로 가져오는건데, 이렇게 문법적으로 해석이 필요한 내용이 종종 나옵니다 🥹
  - ai로 다시 쉽게 풀어 써보겠습니다.
  > "각 객체마다 개별적으로 데이터를 저장하는 대신, 공통 데이터를 하나의 외부 저장소(객체)에 모아두고, 각 객체는 그것을 참조(의존)하는 방식으로 저장할 수 있다."

### 사용법

- 전통적인 플라이웨이트 패턴에는 두가지 개념이 존재합니다.
  - 내재적 intrinsic 상태
    - 객체 자체의 본질적인 정보
    > 책의 경우 동일한 책인경우 항상 동일함
  - 외재적 extrinsic 상태
    - 상황마다 달라지는 부가 정보
    > 책으로 비유해보면 같은 책이어도 사본마다 다름

#### 각 상태에 대한 해석

- 내재적 정보는 객체 내부 메서드에 필요한 것이며, 없으면 동작하지 않는 요소입니다.
  - 동일한 내재적 정보를 지닌 객체를 팩토리 메서드를 사용하여 만들어진 하나의 공유된 객체로 대체할 수 있습니다.
  - 즉, 저장된 내부 데이터 양을 경량화 할 수 있는 내용입니다.
  - 이미 공통 부분으로 인스턴스화된 객체를 재사용하면 되기 때문에 객체의 내내적 정보가 다를 경우에만 새로운 객체 복사본을 생서하면 됩니다.
- 반면 외재적 정보는 제거되어 외부에 저장될 수 있습니다.
  - 외재적 정보를 다룰 때에는 따로 관리자를 사용하게 됩니다.
  > 여기서 관리자란 외재적 데이터를 모아서 관리하는 별도의 구조를 이야기합니다.
  > bookList 배열 + addBook 함수가 관리자 역할을 하는 셈입니다.
  - 관리자는 다양한 방법으로 구현 가능하지만 한 가지 방법만 이야기해본다면, 플라이웨이트 객체와 내재적 상태를 보관하는 중앙 데이터베이스를 관리자로 사용하는 것입니다.
  > books Map이 바로 중앙 데이터베이스입니다.
  > 모든 Book 인스턴스(플라이웨이트 객체)를 한 곳에 모아두고, ISBN으로 조회해서 재사용하는 구조로 이해하면 됩니다.

```
┌───────────────────────────────────────────────┐
│  중앙 저장소 (books Map) ← "관리자"             │
│  ┌──────────────────────────────────────────┐  │
│  │ 'AB123' → Book { title, author, isbn }   │  │  ← 내재적 상태
│  │ 'CD345' → Book { title, author, isbn }   │  │    (공유, 3개만 존재)
│  │ 'EF567' → Book { title, author, isbn }   │  │
│  └──────────────────────────────────────────┘  │
└───────────────────────────────────────────────┘
                      ↑ 참조
┌───────────────────────────────────────────────┐
│  bookList (외부 저장소)                         │
│  ┌──────────────────────────────────────────┐  │
│  │ { ...Book('AB123'), sales:100, avail:✗ } │  │  ← 외재적 상태
│  │ { ...Book('AB123'), sales:50,  avail:✓ } │  │    (개별, 5개 존재)
│  │ { ...Book('CD345'), sales:10,  avail:✓ } │  │
│  │ { ...Book('CD345'), sales:20,  avail:✗ } │  │
│  │ { ...Book('EF567'), sales:20,  avail:✗ } │  │
│  └──────────────────────────────────────────┘  │
└───────────────────────────────────────────────┘
```

> 여기까지의 책 내용을 조금 더 현대적으로 해석하고 풀어서 쓴 내용이 바로 patterns.dev 입니다.

## 강의에서 가져오는 내용

### 플라이웨이트 패턴이란?
- 플라이웨이트의 의미는 fight 스포츠에서 "경량급"을 이야기하는 단어와 의미가 동일합니다.
- GoF 디자인패턴 책에서는 이 패턴을 소개할때 "글자 편집기에서 문자 객체 최적화" 예시를 들며 "메모리 부담을 Flyweight처럼 가볍게" 라는 의미로 명명 하고 있습니다.
  > 뭔말알? 작고 빠르게 작동하는 움직임 -> 가능한 객체를 가볍게 만들자 입니다.
  - 가볍다: 메모리 사용 최소화
  - 공통 데이터(내재적 상태)는 한 번만 저장하고 여러 곳에서 공유
  - 각 객체는 고유한 부분(외재적 상태)만 개별적으로 관리
    > 마치 Flyweight체급 선수처럼 최소한의 무게(메모리)만 가지게 되는 구조입니다.
    > 일반적인 객체를 최홍만 같은 선수로 대입해서 생각하시면 될거같아요. 욘뇨쇼 쵸몬도
    > 각각이 모든 데이터를 독립적으로 소유하는것 자체가 메모리 부하가 크다는 의미죠

### 플라이웨이트의 주요 개념
1. 내재적 상태: 객체의 불변하고 공유 가능한 부분
2. 외재적 상태: 객체의 컨텍스트에 따라 달라지는 부분

- 메모리 사용량 감소
- 객체 생성 비용 감소
- 많은 수의 객체를 효율적으로 관리 가능

#### 활용
- 선임 개발자가 "공통되는 부분을 빼라" 라고 한다면?
  > 대충 미국 잼민이 아하모먼트 짤
  - 플라이웨이트 패턴을 적용해서 내재적 상태를 빼라
    - 내재적 상태: 불변하는 내용과 공유될 수 있는 내용
    - 외재적 상태: 컨텍스트, 호출되는 곳에 따라서 달라지는 내용
  - 도서관에 비유한다면?
    - 일반적인 경우
      - 모든 사람이 자신만의 서재를 가지고, 같은 책을 100명이 원하면 100명을 모두 사야하는 경우가 발생함
      - 즉, 메모리낭비죠?
    - 플라이웨이트 패턴의 경우
      - 공유 가능한 책 (내재적 상태): 책의 제목, 저자, 내용은 변하지 않고 모든 사람이 동일하게 공유함
      - 대출기록 (외재적 상태): 누가 언제 책을 빌렸는지는 각자 다른 데이터를 가짐

#### 실제 개발에서 예시
- 3D 에셋 처리
  - 바위, 나무 같은 3D 모델 에셋을 객체 할달 (내재적 상태)
  - 각 위치 값과 회전값 등은 객체마다 다르게 적용하여 사용 (외재적 상태)
- 리스트 형태, 회원 목록, 게시글 등의 예시
  ```tsx
  const [list, setList] = useState([]);

  const payload = await fetch('');

  // setList(payload) 이 방식을 쓰는건 비효율적입니다. 여러분 이걸 꼭 읽어주세요 제발

  // 공통적으로 사용되는 데이터는 컴포넌트 레벨에서 commonData로 추출
  const { commonData, flyweightedPayload } = extract(payload);

  // 외재적 상태만 경량화된 객체만 set 진행 -> 메모리 절약
  setList(flyweightedPayload);

  return <div>{flyweightedPayload[1].commonData}</div>
  ```
  - 주식 캔들 정보로 다시 생각해보겠습니다.
    - 만개 단위의 데이터를 가져오는 경우 각 객체에 "삼성전자" 라는 프로퍼티가 존재하는 것 자체가 메모리 낭비입니다.
    - 각 객체에서 "삼성전자"의 name 프러퍼티만 추출해주어도 경량화가 진행되는것 입니다.

- 텍스트 포메터 예시
  - 사실 좋은 예시는 아니고 조금 어거지인 너낌이 있는 예시입니다.
  ```tsx
  import React, { memo } from 'react';

  // 포맷팅 함수 - Flyweight의 내재적 상태
  const formatText = (() => {
    const formatters = {
      uppercase: (text) => text.toUpperCase(),
      lowercase: (text) => text.toLowerCase(),
      capitalize: (text) => text.charAt(0).toUpperCase() + text.slice(1),
    };

    return (type, text) => formatters[type]?.(text) || text;
  })();

  // Flyweight TextFormatter 컴포넌트
  const TextFormatter = memo(({ type, children }) => {
    return <span>{formatText(type, children)}</span>;
  });

  // 사용 예시
  const App = () => {
    return (
      <div>
        <TextFormatter type="uppercase">hello world</TextFormatter>
        <TextFormatter type="capitalize">react flyweight</TextFormatter>
        {/* 여러 포매터 인스턴스들이 동일한 포맷팅 함수를 공유 */}
      </div>
    );
  };
  ```
